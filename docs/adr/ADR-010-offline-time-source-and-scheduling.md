## 背景（Context）

Accessify 為**氣隙（air-gapped）軍用內網**部署，執行期無網際網路，因此無法依賴公網 NTP 或外部時間服務。系統需具備「週期重掃 + 排程觸發」能力（見 ROADMAP T601），但精確的 wall-clock 時間在離線環境下可能漂移（宿主機時鐘未校時、容器時區未設定）。

同時排程器選型須與既有架構一致：本產品採 **SQLite(WAL) + in-DB 內嵌任務佇列（無 Redis）**（見 ADR-003），若再引入 `node-cron` 等獨立排程進程，會增加額外依賴與運維面，違背 stability-first（現場服務緩慢、維護成本高）的定位。報表與稽核時間戳需有明確時區語意，避免跨時區誤判。

---

## 評估選項（Options Considered）

### 選項 A：DB 驅動輪詢 + 相對間隔 + 容器 TZ=Asia/Taipei（採用）

- **優點**：與內嵌佇列（ADR-003）一致，**不引入額外進程/依賴**；以相對間隔（相對上次執行時間）觸發，**容忍時鐘漂移**而非依賴精確 wall-clock；容器固定 `TZ=Asia/Taipei`，報表/稽核時間語意明確。
- **缺點**：輪詢有最小粒度延遲（取決於輪詢間隔），非秒級精準排程。
- **風險**：宿主機時鐘嚴重偏移會影響「下次觸發」標示 → 以接內網 NTP 或定期手動校時（runbook）緩解。

### 選項 B：node-cron 獨立排程進程 + 精確 wall-clock

- **優點**：cron 語法直覺、觸發精準到分鐘。
- **缺點**：**新增獨立進程與依賴**，與內嵌佇列雙軌並存（狀態不一致風險）；依賴精確 wall-clock，離線漂移下 cron 時點可能錯置或重複觸發。
- **風險**：與 ADR-003「無額外服務」方向相悖，增加運維面。

### 選項 C：作業系統層 cron（host crontab）觸發容器內指令

- **優點**：複用宿主機既有排程設施。
- **缺點**：排程邏輯逸出應用程式邊界，難以納入應用稽核與站內狀態頁；跨容器/宿主協調脆弱，違反單一可重現交付（ADR-002）。

---

## 決策（Decision）

採 **選項 A：DB 驅動輪詢 + 相對間隔 + 容器 TZ=Asia/Taipei**（依使用者確認）。

1. **排程器選型定案：DB 驅動輪詢**——與內嵌佇列（ADR-003）一致，**不引入 `node-cron` 進程或額外依賴**。
2. **容器 `TZ=Asia/Taipei`**；時間來源 = **宿主機時鐘**（建議現場接內網 NTP 或定期手動校時，列入 runbook）。
3. 排程採**相對間隔**（相對於上次執行，非依賴精確 wall-clock）以**容忍時鐘漂移**。
4. **稽核 / 差異時間戳**記錄時區（Asia/Taipei）並**標註可能漂移**，避免離線環境誤判。
5. 「上次/下次排程」由本地健康/狀態頁顯示（見 ADR-011 / T507）。

---

## 後果（Consequences）

**正面影響：**
- 架構單一（與內嵌佇列同源）、無額外進程；離線時鐘漂移下排程仍可依間隔穩定觸發；報表/稽核時間語意一致（Asia/Taipei）。

**負面影響 / 技術債：**
- 輪詢粒度造成觸發非秒級精準；宿主機時鐘需靠 NTP/手動校時治理（runbook 義務）。

**後續追蹤：**
- [ ] T601：DB 驅動輪詢、相對間隔、`TZ=Asia/Taipei` 實作。
- [ ] runbook：內網 NTP 接入或定期手動校時程序。
- [ ] 狀態頁顯示上次/下次排程（與 T507 整合）。
